# 项目开发规范（前端 + 后端）

**版本**: v2.0.0  
**更新日期**: 2026-05-16  
**适用范围**: GW2 WvW 日志分析系统全项目  
**优先级**: 最高约束，所有代码审查、优化、重构、编写必须严格遵守。

> **核心原则**: 保持原有业务逻辑完全不变，只做规范化、工程化、健壮性优化。  
> **Tailwind CSS 为项目唯一样式标准，禁止手写原生 CSS。**

## 一、全局基础强制约束

### 1.1 编码与文件
1. **UTF-8 编码**: 所有代码、文件、字符必须 UTF-8 编码，绝对禁止乱码、问号、方框、脏字符。
2. **项目结构统一**: 命名、风格、注释、文件组织 100% 高度统一。
3. **配置集中管理**: 环境、主题、请求、常量、枚举、数据库、服务配置全部集中管理，禁止分散配置。
4. **严禁冗余**: 无无用变量、无空函数、无空文件、无未使用导入、无废弃注释、无重复逻辑。
5. **行业规范**: 前端 ESLint + Vue 官方规范；后端 PEP8 + FastAPI 最佳实践。
6. **可运行性**: 代码必须可直接运行、零语法错误、可维护、可扩展。

### 1.2 执行策略（新代码 vs 存量代码）
- **新代码**: 严格执行本规范，超标必须整改后才能合并。
- **存量代码**: 分阶段整改，优先修复安全/权限/性能问题，再逐步优化结构和规范。
- **豁免机制**: 复杂页面因业务逻辑确实难以达标，需提交技术评审纪要，记录原因和整改计划。

---

## 二、代码审查流程（统一前后端）

1. **列出违规点**: 行号 + 问题描述 + 违反规则编号。
2. **给出重构方案**: 说明如何整改，是否需要新建文件。
3. **输出整改代码**: 可直接替换的完整代码片段。
4. **验证**: 整改后必须通过 TypeScript / Python 类型检查及现有测试。
5. **不改动业务逻辑**: 只做分层、规范、格式化、精简、统一异常与响应。

## 三、FastAPI Python 后端规范

### 3.1 行数约束（双轨制）

| 类型 | 目标值 | 红线值 | 超标处理 |
|------|--------|--------|----------|
| 路由文件 | ≤200 行 | ≤300 行 | 超标必须拆分，提交评审纪要 |
| 业务 Service 文件 | ≤400 行 | ≤500 行 | 超标必须拆分 |
| 单文件（任意） | ≤500 行 | ≤600 行 | 超标必须拆分 |

### 3.2 架构约束
1. **严格三层架构**: 路由只写接口入口，禁止写任何业务逻辑；所有业务必须放入 service 层。
2. **函数约束**: 单个函数 ≤80 行，参数 ≤5 个；异步接口必须使用 async/await。
3. **入参校验**: 必须使用 Pydantic 模型校验；禁止硬编码、裸参数、裸响应。
4. **命名规范**: 变量/函数严格小写+下划线；类名大驼峰；接口路径小写+横线。
5. **禁止行为**: 路由层不写数据库操作；不写超大函数；不重复造轮子；不过度拆分。

## 四、Vue3 + PrimeVue 前端规范

### 4.1 行数约束（双轨制）

> **拆分原则**: 禁止过度拆分导致文件泛滥。按语义拆分，<50 行 UI 严禁拆组件；页面私有子组件 3～5 个以内；组件嵌套最多 2 层；单个页面 composables 最多 1～2 个。简单 UI 严禁单独建文件。

| 类型 | 目标值 | 红线值 | 超标处理 |
|------|--------|--------|----------|
| 页面 .vue 文件 | ≤350 行 | ≤500 行 | 超标需拆分，子组件 ≤5 个 |
| 页面 template 部分 | ≤120 行 | ≤180 行 | 提取子组件或抽离 render 函数 |
| 页面 script 部分 | ≤200 行 | ≤280 行 | 抽离 composables |
| 页面 style 部分 | ≤0 行 | ≤0 行 | **禁止页面写 style，全部走 Tailwind** |
| 业务组件 | ≤250 行 | ≤350 行 | 超标必须拆分 |
| 基础组件 | ≤180 行 | ≤250 行 | 超标必须拆分 |

### 4.2 Template 约束
1. **标签嵌套 ≤3 层**（业务允许放宽到 4 层，需注释说明）。
2. **禁止 v-for 与 v-if 同节点**。
   ```vue
   <!-- ❌ 违规 -->
   <div v-for="item in list" v-if="item.show">
   
   <!-- ✅ 规范 -->
   <template v-for="item in list" :key="item.id">
     <div v-if="item.show">
   ```
3. **禁止复杂表达式**: `{{ }}` 和 `:class`/`:style` 中禁止三元嵌套、函数调用、JSON 操作。
   ```vue
   <!-- ❌ 违规 -->
   <div :class="a > 0 ? (b > 0 ? 'x' : 'y') : 'z'">
   <span>{{ Object.entries(map).map(...).join(', ') }}</span>
   
   <!-- ✅ 规范 -->
   <div :class="computedClass">
   <span>{{ formattedMap }}</span>
   ```
4. **v-for 必须带唯一 key**。
5. **禁止行内样式**: 禁止使用 `:style` 和 `style` 属性。
   ```vue
   <!-- ❌ 违规 -->
   <div :style="{ backgroundColor: '#165DFF', width: '500px' }">
   <Column style="min-width: 120px">
   
   <!-- ✅ 规范 -->
   <div class="bg-primary w-[500px]">
   <Column class="min-w-[120px]">
   ```

### 4.3 Script 约束
1. **结构顺序固定**: `import` → `defineProps` → `defineEmits` → `ref/reactive/computed` → `函数定义` → `生命周期`。
2. **Props 约束**: 必须类型+默认+校验；数量 ≤6 个，超标需封装为对象。
   ```ts
   // ❌ 违规：13 个 props
   defineProps<{ a, b, c, d, e, f, g, h, i, j, k, l, m }>()
   
   // ✅ 规范：≤6 个，或封装对象
   defineProps<{ player: PlayerInfo; options: PlayerOptions }>()
   ```
3. **函数约束**: 单个函数 ≤50 行；复杂逻辑抽 computed / composables。
4. **禁止硬编码**: 所有常量、枚举、配置统一抽取到 `constants/` 或 `config/`。
5. **TypeScript 规范**:
   - 禁止 `any`，优先使用 `unknown` + 类型守卫。
   - 禁止滥用 `as` 类型断言。
   - 接口命名使用大驼峰，属性使用 camelCase。
   - 优先类型推导，冗余类型注解应删除。

### 4.4 CSS 约束（Tailwind CSS 唯一标准）
1. **Tailwind CSS 为项目唯一样式标准，禁止手写原生 CSS。**
2. ** scoped 可选**: Vue 单文件组件中 `style` 块非必需；如有，必须 `scoped`。
3. **严禁** `!important` 和 `:deep()`，特殊情况需注释说明原因。
4. **颜色/间距/圆角/阴影**: 必须使用设计系统 token（CSS 变量或 Tailwind 配置）。
   ```vue
   <!-- ❌ 违规 -->
   <div class="bg-blue-600 text-white p-4 rounded-lg">
   
   <!-- ✅ 规范 -->
   <div class="bg-primary text-surface-0 p-card-radius rounded-card">
   ```
5. **类名遵循 BEM**: 自定义类名（非 Tailwind）使用 BEM，如 `.card__header--active`。

### 4.5 组件约束
1. **页面禁止直接使用 PrimeVue 原生标签**，必须使用项目二次封装的 Base 组件。
   ```vue
   <!-- ❌ 违规 -->
   <Button label="保存" />
   <Tag value="标签" />
   
   <!-- ✅ 规范 -->
   <BaseButton label="保存" />
   <BaseTag value="标签" />
   ```
2. **单一职责**: 一个组件只做一件事。
3. **重复 UI 必须抽取**: 同一 UI 片段在 2 个以上文件出现，必须抽取为公共组件。
4. **组件嵌套 ≤2 层**。

### 4.6 分层约束
1. **接口放 api 层**: 所有 API 调用集中在 `services/` 目录，通过 `services/index.ts` 统一暴露。
2. **业务逻辑抽 composables**: 页面不做业务逻辑，只做组装。
3. **状态走 Pinia**: 跨页面共享状态必须使用 Pinia Store；页面本地状态可用 `ref`/`reactive`。
4. **工具放 utils**: 纯工具函数放入 `utils/`，业务相关工具放入对应模块目录。

### 4.7 权限控制规范（新增）
1. **统一使用 `v-permission` 指令**，废弃 `canWrite` prop 和 `isAdmin` prop 模式。
   ```vue
   <!-- ✅ 规范 -->
   <BaseButton v-permission="'delete'" @click="handleDelete" />
   <BaseButton v-permission.disabled="'write'" @click="handleEdit" />
   ```
2. **权限粒度**:
   - `v-permission="'read'"` → 读权限
   - `v-permission="'write'"` → 写权限（含编辑、创建）
   - `v-permission="'delete'"` → 删除权限
   - `v-permission="'admin'"` → 仅超级管理员
   - `v-permission="'logged_in'"` → 需认证（操作员或管理员）
3. **所有交互按钮必须绑定权限**，游客只能查看，不能操作。
4. **路由守卫**: 系统管理页面统一要求 `permissions: ['write']`。

### 4.8 国际化规范（新增）
1. **所有用户可见文本必须走 `vue-i18n`**，禁止硬编码中文字符串。
   ```vue
   <!-- ❌ 违规 -->
   <span>数据看板</span>
   <BaseButton label="保存" />
   
   <!-- ✅ 规范 -->
   <span>{{ $t('dashboard.title') }}</span>
   <BaseButton :label="$t('common.save')" />
   ```
2. **新增功能必须同时添加中英文语言包**。
3. **存量硬编码**: 分阶段迁移，新代码零容忍。

### 4.9 安全规范（新增）
1. **XSS 防护**: 禁止 `v-html`；如必须使用，输入必须经过 DOMPurify 消毒。
2. **CSRF 防护**: API 请求必须携带 CSRF Token。
3. **敏感数据**: 禁止在代码中硬编码密码、API Key、Token。
4. **权限最小化**: 前端不暴露无权限的操作入口（即使后端有校验）。

### 4.10 性能与健壮性
1. **异步请求必须 try/catch**，统一错误处理。
2. **高频事件防抖节流**（搜索输入、窗口 resize、滚动）。
3. **大列表必须分页/虚拟滚动**（>100 条数据）。
4. **列表渲染优化**: 避免在 `v-for` 模板中直接调用函数（如 `getProfessionName`），应在数据准备阶段预计算。

### 4.11 注释规范
1. 公共组件、方法、复杂逻辑必须注释。
2. 注释使用中文，描述"做了什么"而非"怎么做的"。
3. 临时代码/ workaround 必须标注 `// TODO:` 或 `// FIXME:` 并附带说明和日期。

## 五、Git 工作流规范（新增）

1. **分支模型**:
   - `master`: 生产环境，保护分支。
   - `test`: 测试环境，开发集成。
   - `feature/*`: 功能分支，从 `test` 切出，完成后合并回 `test`。
2. **提交规范**: 使用 Conventional Commits。
   ```
   <type>(<scope>): <description>
   
   [optional body]
   [optional footer]
   ```
3. **Code Review**: 所有合并必须至少 1 人 Review，重点检查权限、安全、行数、硬编码。

## 六、设计系统规范（新增）

1. **颜色**: 禁止硬编码 hex/rgb，必须使用 CSS 变量或 Tailwind 配置。
   ```css
   /* 项目 token */
   --color-primary: #165DFF;
   --color-surface-0: #ffffff;
   --color-surface-900: #0a0a0a;
   ```
2. **间距**: 使用标准间距 token（4px 基数的倍数）。
3. **圆角**: 使用标准圆角 token（如 `--radius-card: 12px`）。
4. **响应式**: 移动端优先，断点统一为 `sm:640px md:768px lg:1024px xl:1280px`。

---

## 七、输出格式（代码审查报告）

审查报告必须包含以下 4 部分：

1. **违规问题清单**: 文件路径 + 行号 + 违反规则编号 + 问题描述。
2. **优化重构方案**: 如何整改，是否需要新建文件/组件/composables。
3. **整改后完整代码**: 可直接替换的代码片段。
4. **新增依赖文件代码**: 如有新建文件，一并给出完整代码。

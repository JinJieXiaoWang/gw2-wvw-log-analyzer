# 一、40G 存储空间数据容量分析

> **分析日期**: 2026-05-05
> **目标存储**: MySQL 8.0 (InnoDB) / SQLite / PostgreSQL
> **可用空间**: 40 GB
> **参考样本**: `tests/20260426-220412.zevtc` (367秒 WvW 详细战斗日志)

## 1. 数据样本基线分析

### 1.1 单条日志各形态文件大小

| 数据形态 | 文件 | 大小 (MB) | 占比 | 说明 |
|----------|------|-----------|------|------|
| **原始 ZEVTC** | `.zevtc` | ~8.39 | 基准 | ZIP压缩后的arcdps原始日志 |
| **EI 完整 JSON** | `_kill.json` | ~195.09 | 2325% | Elite Insights解析后的完整数据 |
| **简化 JSON** | `parsed_ei_format.json` | ~0.61 | 7.3% | 抽取后的核心聚合数据 |
| **CSV 导出** | `_kill.csv` | ~0.05 | 0.6% | 仅含基础DPS排行 |

> **注意**: v3.0 起，系统不再存储完整的 EI JSON（195MB），而是选择性同步到数据库表中。

### 1.2 当前数据库存储结构（v3.0）

单条日志在数据库中的占用估算：

```
fights (~1行)                  : ~1 KB
fight_stats (~51行)            : 51 × 0.5 KB = 25.5 KB
ei_player (~51行)              : 51 × 2 KB = 102 KB  (JSON列存储完整玩家数据)
ei_target (~83行)              : 83 × 1 KB = 83 KB
ei_skill_map (~727行)          : 727 × 0.3 KB = 218 KB
ei_phase (~5行)                : 5 × 0.5 KB = 2.5 KB
索引开销 (×1.5)                : ×1.5
─────────────────────────────────────────────────
单条合计                        : ~650 KB ≈ 0.63 MB
```

基于一个活跃WvW公会（50人规模，每晚团战2-3小时）的使用场景：

| 场景 | 每场战斗时长 | 每场产生日志数 | 每晚场次 | 日产量 |
|------|-------------|---------------|----------|--------|
| **轻度和弦** (休闲公会) | 2-5分钟 | 15-30条 | 5-10场 | ~200条/日 |
| **中度和弦** (活跃公会) | 5-10分钟 | 30-60条 | 10-20场 | ~800条/日 |
| **重度和弦** (竞技公会) | 10-15分钟 | 60-100条 | 20-30场 | ~2,000条/日 |
| **极限和弦** (服务器主力) | 15-30分钟 | 100-200条 | 30-50场 | ~5,000条/日 |

**本文以中度和弦（800条/日）为主要分析基准。**

## 2. 存储模式对比分析

### 2.1 三种存储模式的空间占用

#### 模式 A: 仅保留原始 ZEVTC 文件

```
单条: 8.39 MB
每日: 800 × 8.39 MB = 6,712 MB ≈ 6.55 GB
40G 可用天数: 40 / 6.55 ≈ 6.1 天
```

| 指标 | 数值 |
|------|------|
| 单条占用 | 8.39 MB |
| 日总量 | 6.55 GB |
| **40G 最大天数** | **~6 天** |
| 优点 | 保留原始数据，可重新解析 |
| 缺点 | 无法直接查询，空间消耗大 |

#### 模式 B: 仅数据库存储（v3.0 推荐）

基于当前 `fights` + `fight_stats` + `ei_*` 表结构：

```
单条: ~0.63 MB
每日: 800 × 0.63 MB = 504 MB
40G 可用天数: 40,000 / 504 ≈ 79 天
```

| 指标 | 数值 |
|------|------|
| 单条占用 | 0.63 MB |
| 日总量 | 504 MB |
| **40G 最大天数** | **~79 天（约2.6个月）** |
| 优点 | 可查询、空间可控、无需保留原始文件 |
| 缺点 | 丢失原始事件流，无法重新用新版本 EI 解析 |

#### 模式 C: 混合存储（数据库 + 压缩原始文件）

保留数据库聚合数据 + 原始 ZEVTC 文件（gzip 压缩后约 8.39MB）：

```
单条: 0.63 + 8.39 = 9.02 MB
每日: 800 × 9.02 MB = 7,216 MB ≈ 7.05 GB
40G 可用天数: 40 / 7.05 ≈ 5.7 天
```

| 指标 | 数值 |
|------|------|
| 单条占用 | 9.02 MB |
| 日总量 | 7.05 GB |
| **40G 最大天数** | **~5.7 天** |
| 优点 | 保留原始数据，同时可查询 |
| 缺点 | 空间消耗接近纯 ZEVTC 存储 |

### 2.2 三种模式对比总览

| 模式 | 单条(MB) | 日总量(GB) | 40G天数 | 功能完整度 | 查询性能 | 推荐度 |
|------|---------|-----------|---------|-----------|---------|--------|
| A: 仅ZEVTC | 8.39 | 6.55 | **6天** | ⭐ (仅存储) | ❌ 不可查询 | ⭐⭐ |
| B: 仅数据库 | 0.63 | 0.49 | **79天** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| C: 混合存储 | 9.02 | 7.05 | **5.7天** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |

## 3. 推荐存储策略

### 3.1 核心原则

v3.0 推荐的存储策略：**数据库存储为主，可选保留压缩原始文件**

```
┌─────────────────────────────────────────────────────────────┐
│                    应用层 (GW2-WvW-Log-Analyzer)             │
└─────────────────────────────────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   热数据层   │    │   温数据层   │    │   冷数据层   │
│  (数据库)    │    │  (数据库)    │    │  (文件系统)  │
│  最近30天    │    │  31-180天    │    │  180天+      │
│  完整JSON    │    │  仅聚合统计   │    │  仅元数据    │
└──────────────┘    └──────────────┘    └──────────────┘
        │                    │                    │
        ▼                    ▼                    ▼
   fights + ei_*        fights + fight_stats   evtc_log 元数据
   (完整EI同步)         (核心聚合)             + 可选 gzip 文件
```

### 3.2 分层存储配置

## 4. 数据清理策略

### 4.1 自动清理机制

系统通过 `AUTO_CLEANUP_ENABLED` 启用自动清理：

```python
# 清理策略配置
cleanup_policy = {
    # 热数据：保留完整 ei_* 表（30天）
    "hot_retention_days": 30,
    
    # 温数据：仅保留 fights + fight_stats（31-180天）
    "warm_retention_days": 180,
    
    # 冷数据：仅保留 evtc_log 元数据（180天+）
    "cold_retention_days": 365,
    
    # 清理时间窗口
    "cleanup_cron": "0 3 * * *",  # 每天凌晨3点
}
```

### 4.2 三级清理策略

#### 热数据清理（每日执行）

保留完整 `ei_*` 表（30天内）：

```sql
-- 删除 30 天前的 ei_* 详细数据
DELETE FROM ei_player WHERE log_id IN (
    SELECT log_id FROM evtc_log WHERE upload_time < DATE_SUB(NOW(), INTERVAL 30 DAY)
);
DELETE FROM ei_target WHERE log_id IN (...);
DELETE FROM ei_skill_map WHERE log_id IN (...);
DELETE FROM ei_phase WHERE log_id IN (...);
```

#### 温数据清理（每日执行）

仅保留聚合统计（31-180天）：

```sql
-- 保留 fights + fight_stats，删除 ei_*
-- （同上，ei_* 在 30 天后已清理，此处确保完整性）
```

#### 冷数据清理（每月执行）

仅保留 `evtc_log` 元数据（180天+）：

```sql
-- 删除 fights 和 fight_stats
DELETE FROM fight_stats WHERE fight_id IN (
    SELECT id FROM fights WHERE log_id IN (
        SELECT log_id FROM evtc_log WHERE upload_time < DATE_SUB(NOW(), INTERVAL 180 DAY)
    )
);
DELETE FROM fights WHERE log_id IN (
    SELECT log_id FROM evtc_log WHERE upload_time < DATE_SUB(NOW(), INTERVAL 180 DAY)
);

-- 保留 evtc_log 元数据（用于去重判定）
```

### 4.3 文件清理

```python
# 清理过期的 .zevtc 原始文件
if not KEEP_RAW_FILE_AFTER_PARSE:
    # 解析成功后立即删除
    os.remove(file_path)
else:
    # 按保留策略清理
    if file_age > FILE_RETENTION_DAYS:
        os.remove(file_path)
```

## 5. 重复导入判定机制

### 5.1 核心问题

数据清理之后，再次导入同一文件，是**新增**还是**更新**？

**场景分析**:
- 场景A: 文件之前导入过，数据仍在热数据层（30天内）→ **拒绝/提示已存在**
- 场景B: 文件之前导入过，聚合数据保留但详细数据已清理 → **用户选择：跳过/重新解析更新**
- 场景C: 文件是新的，从未导入过 → **正常新增**
- 场景D: 文件内容相同但文件名不同 → **通过 SHA-256 指纹判定为重复**

### 5.2 判定体系设计

#### 三层判定机制

```
第一层: 文件指纹判定 (SHA-256)
├── 指纹存在于 evtc_log.file_sha256 ?
│   ├── 是 → 进入第二层
│   └── 否 → 新文件 → 新增流程
│
第二层: 数据存在性判定
├── 指纹对应的 log_id 在数据库中存在 ?
│   ├── 是 (有 ei_* 数据) → 返回 "已存在，数据完整"
│   ├── 是 (仅 fights/fight_stats) → 返回 "已存在，仅聚合数据"
│   └── 是 (仅 evtc_log 元数据) → 进入第三层
│
第三层: 用户意图判定
├── 数据完整 → 仅允许"查看现有数据"或"强制重新解析(更新)"
├── 仅聚合数据 → 提供选项:
│   ├── "跳过" (保留现有记录)
│   ├── "重新解析-更新" (删除旧数据，重新解析入库)
│   └── "重新解析-追加版本" (保留历史版本)
└── 数据已彻底删除 → 视为新导入，走新增流程
```

### 5.3 指纹持久化

`evtc_log.file_sha256` 字段**永不删除**，仅用于去重判定：

```sql
-- 即使所有战斗数据已清理，evtc_log 行保留
SELECT file_sha256, upload_time, parse_status 
FROM evtc_log 
WHERE file_sha256 = ?;
```

## 6. 存储优化建议

### 6.1 MySQL 优化

```sql
-- 启用 InnoDB 压缩（可选）
ALTER TABLE ei_player ROW_FORMAT=COMPRESSED;
ALTER TABLE ei_target ROW_FORMAT=COMPRESSED;

-- 对 JSON 列建立虚拟列索引（MySQL 8.0+）
ALTER TABLE ei_player ADD COLUMN profession_idx VARCHAR(50) 
    GENERATED ALWAYS AS (profession) STORED;
CREATE INDEX idx_ei_player_prof ON ei_player(profession_idx);
```

### 6.2 PostgreSQL 优化

```sql
-- 对 JSONB 列建立 GIN 索引
CREATE INDEX idx_ei_player_dps ON ei_player USING GIN (dps_all_json);

-- 表分区（按 log_id 范围）
CREATE TABLE ei_player_partitioned (LIKE ei_player) 
    PARTITION BY RANGE (log_id);
```

### 6.3 SQLite 优化

```sql
-- 启用 WAL 模式
PRAGMA journal_mode = WAL;

-- 增加缓存大小
PRAGMA cache_size = -65536;  -- 64MB
```

## 8. 总结

| 问题 | 答案 |
|------|------|
| **40G 纯数据库存储能存多久？** | 活跃公会（800条/日）: **~81 天** |
| **如何延长存储周期？** | 缩短热数据保留期（30天→7天），仅保留聚合数据可延长至 **~300 天** |
| **单条日志数据库占用？** | ~0.63 MB（含 fights + fight_stats + ei_* 全量同步） |
| **重复导入如何判定？** | SHA-256 指纹 + 三级判定（指纹→数据存在性→用户意图） |
| **清理后重新导入是新增还是更新？** | **更新** — 删除旧数据后重新解析入库，log_id 不变 |
| **evtc_log 元数据会无限增长吗？** | 是，但极慢 — 800条/日×365天×200字节 ≈ **58MB/年**，可忽略 |
